## Бенчмарк semantic_query на 10 000 записей

```
Подготовка 10 000 эмбеддингов: 2635.6s (~43.9 min, ~264ms/embed)
Загрузка 10 000 векторов в LanceDB: 0.58s
semantic_query over 200 samples (n=10000):
  mean = 247.9 ms
  p50  = 247.7 ms
  p95  = 273.3 ms
  p99  = 289.3 ms
SLA p99 ≤ 500.0 ms → OK
```

Среднее время ~248 мс распределяется примерно так:
- ~210 мс на эмбеддинг запроса через Ollama (синхронный POST, CPU);
- ~30 мс на brute-force поиск в LanceDB (10 000 × 1024 float32);
- ~5 мс на сериализацию ответа.

Эмбеддинг — доминирующая стоимость. Для агрессивного ускорения понадобилось бы кешировать эмбеддинги одинаковых запросов, либо переключить Ollama на GPU. В дизайне этого не требуется — SLA 500 мс выполнен с запасом.

## Фиксы на ходу

- `lancedb.list_tables()` в версии 0.30.2 возвращает `ListTablesResponse` (pydantic-модель с полем `.tables`), а не `list[str]`. Добавил ветвление с `hasattr(raw, "tables")`.
- `OLLAMA_HOST` может содержать порт (`127.0.0.1:11434`). Парсинг через `urllib.parse.urlsplit`.

## Решения

- **Без явного IVF_HNSW-индекса на Typical_Volume.** LanceDB 0.30.2 с brute-force cosine на 10 000 × 1024 float32 укладывается в 30 мс, что меньше стоимости одного эмбеддинга. Создание индекса требует минимум 256 записей для осмысленной тренировки и добавляет overhead на запись. Оставил как опцию для будущего через `tools/sml/embedding_engine.py::EmbeddingStore`.
- **Brute-force лучше прячет нестабильности метрики.** При малом числе записей ANN-индексы могут возвращать неоптимальные результаты; brute-force детерминирован, что важно для property-теста P8.
- **Кеш эмбеддингов для бенчмарка.** Первый прогон 10 000 × ~264 мс = 44 минуты. Повторный прогон с кешем из `var/sml/bench-embed-cache.json` займёт несколько секунд — только замеры поиска, без перерасчёта.

## Риски и ограничения

- **Эмбеддинг — узкое место.** ~260 мс на одно обращение к `bge-m3` на CPU. Для частых запросов можно ввести LRU-кеш в `OllamaEmbedder` (по хешу текста). Отложено до Этапа 7.
- **Первый запуск бенчмарка дорогой.** 44 минуты — это стоимость первой подготовки кеша. Повторные запуски — секунды. В CI это неприемлемо; в CI бенчмарк запускать только с флагом `--quick` (n=100).
- **LanceDB pydantic-модели.** API 0.30.2 ещё не стабилен — `list_tables()` вернул не то, что в старых туториалах. Если обновимся, возможны регрессии — покрыто тестом `test_store_upsert_and_count`.
