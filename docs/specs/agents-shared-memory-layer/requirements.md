# Документ требований

## Введение

Слой общей памяти агентов (Shared_Memory_Layer, далее SML) — это единый stateful-слой памяти, к которому подключаются Codex, Cursor, Kiro и любые будущие AI-агенты инфраструктуры `D:\AionUi-Paperclip`. SML заменяет текущий grep-подход `aion-file-memory` как основной канал чтения и записи знаний, но не заменяет файловый источник истины в `docs/` и `AGENTS.md`.

Цель SML — обеспечить единое состояние памяти между агентами, его стойкость к перезагрузкам и смене сессий, накопление фактов и решений, семантический и временной поиск, а также взаимозаменяемость агентов. Документ фиксирует только требования к поведению слоя. Выбор конкретной технологии (Letta, Graphiti/Zep, Basic Memory, Mem0 или их композиция) производится на этапе дизайна.

## Глоссарий

- **Shared_Memory_Layer (SML)**: единый stateful-сервис памяти, к которому подключаются все агенты инфраструктуры `D:\AionUi-Paperclip` через MCP.
- **Agent**: программный агент (Codex CLI, Cursor, Kiro или любой будущий), который читает и пишет память через MCP.
- **Memory_Record**: единица хранения SML с типом (факт, предпочтение, решение, запись журнала, задача, связь, ограничение, таймлайн-событие), текстом, автором, меткой времени, тегами и ссылками на исходные файлы в `docs/`.
- **File_Memory**: существующая файловая память — `AGENTS.md`, `docs/current-context.md`, `docs/tasks.md`, `docs/decisions.md`, `docs/agent-log/`, `docs/handoffs/`, `docs/memory/layers/`, `docs/context-packs/context-pack-latest.md`.
- **MCP_Protocol**: протокол Model Context Protocol, через который агенты вызывают инструменты SML.
- **Session**: один запущенный процесс конкретного агента от старта до завершения.
- **Fallback_Mode**: режим работы агента, при котором SML недоступен, и агент читает только File_Memory.
- **Commit**: успешное завершение операции записи в SML (все последующие операции чтения возвращают записанное значение).
- **Semantic_Query**: запрос к SML по смыслу на русском языке, возвращающий Memory_Record по семантической близости, а не только по подстроке.
- **Temporal_Query**: запрос к SML, учитывающий эволюцию Memory_Record во времени (актуальность, устаревание, суперседирование).
- **Operation_Log**: лог всех операций чтения и записи SML с меткой времени, именем агента и идентификатором Memory_Record.
- **Typical_Volume**: типичный объём базы SML для этого проекта — до 10 000 Memory_Record и до 200 МБ индекса.

## Требования

### Требование 1: Подключение агентов по MCP

**User Story:** Как владелец инфраструктуры `D:\AionUi-Paperclip`, я хочу, чтобы Codex, Cursor, Kiro и будущие агенты подключались к SML через единый протокол, чтобы не писать клиентский код для каждого агента отдельно.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL предоставлять через MCP_Protocol набор операций чтения и записи памяти, включающий как минимум операции поиска, получения и добавления записей памяти, с единым контрактом входных и выходных данных для любого Agent.
2. THE Shared_Memory_Layer SHALL публиковать список MCP-инструментов, совместимых с конфигурациями MCP в `~/.codex/config.toml`, `.cursor/mcp.json` и `.kiro/settings/mcp.json`, с одинаковыми именами и сигнатурами инструментов во всех трёх конфигурациях.
3. WHEN Agent запускается и конфигурация MCP указывает на Shared_Memory_Layer, THE Shared_Memory_Layer SHALL принимать подключение Agent без запроса подтверждения у пользователя и завершать установление соединения не дольше чем за 5 секунд.
4. WHERE Agent подключён, THE Shared_Memory_Layer SHALL предоставлять Agent одинаковый набор MCP-инструментов с идентичными именами, параметрами и форматами ответа независимо от типа Agent.
5. IF Agent использует newline-stdio транспорт MCP, THEN THE Shared_Memory_Layer SHALL обрабатывать запросы Agent и отправлять ответы по тому же транспорту, сохраняя соответствие между идентификатором запроса и идентификатором ответа.
6. IF подключение Agent к Shared_Memory_Layer завершается ошибкой транспорта или несовместимостью версии MCP_Protocol, THEN THE Shared_Memory_Layer SHALL отклонять подключение с сообщением об ошибке, указывающим причину отказа, и сохранять уже принятые подключения других Agent без прерывания.

### Требование 2: Единое состояние памяти между агентами

**User Story:** Как пользователь, работающий поочерёдно в Codex, Cursor и Kiro, я хочу, чтобы изменение, сделанное одним агентом, было видно остальным, чтобы мне не приходилось вручную переносить контекст.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL хранить все Memory_Record в едином хранилище, общем для всех подключённых Agent, и обеспечивать идентичный набор Memory_Record при запросе полного списка от любого Agent.
2. WHEN Agent успешно выполняет Commit Memory_Record, THE Shared_Memory_Layer SHALL сделать этот Memory_Record доступным для чтения другим Agent не позднее 2 секунд после завершения Commit при условии стабильного сетевого соединения.
3. WHEN два или более Agent одновременно отправляют операции записи одного и того же Memory_Record в интервале до 500 миллисекунд, THE Shared_Memory_Layer SHALL сериализовать операции и сохранить все версии с уникальными метками времени с точностью до миллисекунды и именами авторов-Agent.
4. WHILE между двумя запросами чтения одного и того же Memory_Record не зафиксировано ни одной операции Commit, THE Shared_Memory_Layer SHALL возвращать идентичный результат любому Agent, включая одинаковый идентификатор версии и одинаковое содержимое.
5. IF Commit Memory_Record завершается с ошибкой, THEN THE Shared_Memory_Layer SHALL отклонить Memory_Record, не публиковать его другим Agent и вернуть инициировавшему Agent ответ с индикацией отказа и причиной категории ошибки, сохранив предыдущее состояние Memory_Record без изменений.
6. IF Agent запрашивает чтение Memory_Record, отсутствующего в едином хранилище, THEN THE Shared_Memory_Layer SHALL вернуть ответ с индикацией отсутствия Memory_Record в течение 2 секунд с момента получения запроса.

### Требование 3: Стойкость памяти между сессиями

**User Story:** Как пользователь, я хочу, чтобы накопленные знания переживали закрытие чата, перезагрузку машины и смену сессии, чтобы не терять контекст между днями работы.

#### Acceptance Criteria

1. WHEN Agent выполняет Commit для Memory_Record, THE Shared_Memory_Layer SHALL записывать Memory_Record на постоянный локальный диск и возвращать успешный ответ о Commit только после завершения записи, не позднее 5 секунд с момента получения запроса Commit.
2. WHEN Session агента завершается штатно или аварийно, THE Shared_Memory_Layer SHALL сохранять все Memory_Record, получившие успешный ответ о Commit до завершения Session, и предоставлять эти Memory_Record любой новой Session того же или другого Agent без потерь и без изменения содержимого.
3. WHEN Shared_Memory_Layer запускается после перезагрузки машины или после штатной остановки, THE Shared_Memory_Layer SHALL предоставлять все Memory_Record, получившие успешный ответ о Commit до остановки, в течение 30 секунд с момента запуска и без изменения содержимого.
4. IF процесс Shared_Memory_Layer аварийно завершается до возврата успешного ответа о Commit для Memory_Record, THEN THE Shared_Memory_Layer SHALL после перезапуска сохранять все Memory_Record, получившие успешный ответ о Commit до сбоя, и SHALL отклонять частично записанный Memory_Record, не получивший успешного ответа о Commit, без повреждения ранее сохранённых Memory_Record.
5. IF чтение или запись Memory_Record на постоянный локальный диск завершается ошибкой ввода-вывода, THEN THE Shared_Memory_Layer SHALL возвращать Agent ответ об ошибке Commit с указанием причины сбоя и SHALL сохранять все ранее зафиксированные Memory_Record в неизменном виде.

### Требование 4: Накопление знаний и самообучение

**User Story:** Как пользователь, я хочу, чтобы слой памяти накапливал факты, предпочтения, решения и связи между задачами по мере работы агентов, чтобы со временем контекст становился полнее без ручного структурирования.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL поддерживать типы Memory_Record: факт, предпочтение, решение, запись журнала, задача, связь между задачами, ограничение, таймлайн-событие.
2. WHEN Agent вызывает операцию записи с типом Memory_Record из перечня поддерживаемых типов и обязательными полями (тип, содержимое длиной от 1 до 10000 символов, имя Agent-автора длиной от 1 до 128 символов), THE Shared_Memory_Layer SHALL сохранить Memory_Record, присвоить уникальный идентификатор, метку времени в формате ISO 8601 с точностью до миллисекунд и зафиксировать имя Agent-автора, и вернуть присвоенный идентификатор за время не более 2 секунд.
3. IF Agent вызывает операцию записи с неподдерживаемым типом Memory_Record, с отсутствующими обязательными полями или с содержимым, нарушающим установленные границы длины, THEN THE Shared_Memory_Layer SHALL отклонить операцию, не создавать Memory_Record, не изменять состояние хранилища и вернуть признак ошибки с указанием причины отклонения.
4. WHEN Agent добавляет Memory_Record со ссылкой на идентификатор существующего Memory_Record, THE Shared_Memory_Layer SHALL сохранить связь между указанными Memory_Record и включать связанный Memory_Record в результаты запросов по исходному Memory_Record.
5. IF Agent добавляет Memory_Record со ссылкой на идентификатор, не соответствующий ни одному существующему Memory_Record, THEN THE Shared_Memory_Layer SHALL отклонить операцию, не создавать Memory_Record и не создавать связь, и вернуть признак ошибки о несуществующем идентификаторе.
6. THE Shared_Memory_Layer SHALL увеличивать объём хранимых Memory_Record монотонно: ни один подтверждённый Memory_Record не удаляется и не изменяется автоматически без явной операции удаления или операции суперседирования, инициированной Agent.

### Требование 5: Семантический поиск на русском языке

**User Story:** Как пользователь, я хочу, чтобы агент находил релевантные воспоминания по смыслу запроса на русском, чтобы не зависеть от точного совпадения подстроки.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL предоставлять операцию Semantic_Query, принимающую текстовый запрос на русском языке длиной от 1 до 1000 символов.
2. IF текст Semantic_Query пуст, состоит только из пробельных символов или превышает 1000 символов, THEN THE Shared_Memory_Layer SHALL отклонить Semantic_Query с признаком ошибки валидации и не возвращать Memory_Record.
3. WHEN Agent выполняет Semantic_Query при Typical_Volume, THE Shared_Memory_Layer SHALL возвращать не более 50 Memory_Record, отсортированных по убыванию семантической близости к запросу, в течение 2 секунд.
4. THE Shared_Memory_Layer SHALL возвращать в результатах Semantic_Query значение метрики релевантности для каждого Memory_Record в диапазоне от 0.0 до 1.0 с точностью не менее трёх знаков после запятой.
5. WHEN Semantic_Query содержит русскоязычный термин, семантически совпадающий с Memory_Record, в котором использован синоним или перефразирование, THE Shared_Memory_Layer SHALL включить этот Memory_Record в первые 10 результатов при Typical_Volume.
6. IF ни один Memory_Record не достигает порога семантической близости 0.5, THEN THE Shared_Memory_Layer SHALL вернуть пустой список результатов Semantic_Query с признаком отсутствия совпадений.

### Требование 6: Учёт времени и актуальности

**User Story:** Как пользователь, я хочу, чтобы память различала устаревшие и актуальные факты, чтобы агенты опирались на последние решения и не цитировали отменённые.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL хранить для каждого Memory_Record метку времени создания в формате ISO 8601 с точностью до миллисекунд и в часовом поясе UTC, метку времени последнего изменения в том же формате и флаг актуальности со значениями «актуален» или «неактуален».
2. WHEN Agent фиксирует Memory_Record, который суперседирует один или несколько существующих Memory_Record, THE Shared_Memory_Layer SHALL в рамках одной атомарной операции пометить каждый предыдущий Memory_Record как неактуальный, обновить метку времени последнего изменения предыдущего Memory_Record и сохранить ссылку на суперседирующий Memory_Record.
3. IF суперседирующий Memory_Record ссылается на несуществующий или уже удалённый предыдущий Memory_Record, THEN THE Shared_Memory_Layer SHALL отклонить фиксацию суперседирующего Memory_Record, не изменять состояние существующих Memory_Record и вернуть Agent сообщение об ошибке с указанием идентификатора отсутствующего Memory_Record.
4. WHEN Agent выполняет Temporal_Query с указанной меткой времени, THE Shared_Memory_Layer SHALL возвращать набор Memory_Record в том состоянии флага актуальности, которое действовало на указанную метку времени, включая Memory_Record, созданные до указанной метки времени и не удалённые до неё.
5. IF Temporal_Query содержит метку времени в будущем относительно текущего серверного времени или метку времени раньше момента создания первого Memory_Record, THEN THE Shared_Memory_Layer SHALL отклонить Temporal_Query и вернуть Agent сообщение об ошибке с указанием допустимого диапазона меток времени.
6. WHEN Agent выполняет Semantic_Query без параметра включения неактуальных Memory_Record, THE Shared_Memory_Layer SHALL возвращать только Memory_Record с флагом актуальности «актуален» и исключать Memory_Record с флагом «неактуален» из результатов.
7. WHERE Agent в Semantic_Query явно указывает параметр включения неактуальных Memory_Record, THE Shared_Memory_Layer SHALL включать Memory_Record с флагом «неактуален» в результаты, сопровождать каждый такой Memory_Record пометкой неактуальности и ссылкой на суперседирующий Memory_Record, если такая ссылка существует.

### Требование 7: Взаимозаменяемость агентов и быстрый вход в контекст

**User Story:** Как пользователь, я хочу, чтобы новый или заменяющий агент получал полный контекст проекта без ручного объяснения, чтобы перейти с Codex на Cursor или Kiro без потерь.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL предоставлять операцию получения стартового контекстного пакета для Agent, возвращающую результат не позднее 5 секунд с момента вызова.
2. WHEN Agent запрашивает стартовый контекстный пакет, THE Shared_Memory_Layer SHALL вернуть набор Memory_Record, покрывающий разделы: природа проекта, действующие решения, активные задачи, предпочтения пользователя, ограничения, последние 10 записей журнала, при этом каждый Memory_Record SHALL быть обновлён не позднее 30 дней до момента вызова.
3. THE Shared_Memory_Layer SHALL возвращать стартовый контекстный пакет в единообразной структуре с фиксированным составом и порядком разделов, одинаковым для любого типа Agent.
4. WHEN Agent подключается впервые и вызывает операцию получения стартового контекстного пакета, THE Shared_Memory_Layer SHALL вернуть тот же состав и порядок разделов, что и для уже работавшего Agent, без учёта предыдущей истории вызовов Agent.
5. IF при формировании стартового контекстного пакета хотя бы один из обязательных разделов отсутствует или не содержит ни одного Memory_Record, THEN THE Shared_Memory_Layer SHALL вернуть стартовый контекстный пакет с явным признаком пустого раздела для каждого такого раздела и с индикацией неполноты пакета, сохраняя остальные разделы без изменений.
6. IF операция получения стартового контекстного пакета завершается сбоем, THEN THE Shared_Memory_Layer SHALL вернуть Agent индикацию ошибки с указанием категории сбоя и SHALL не изменять состояние Memory_Record.

### Требование 8: Сосуществование с файловой памятью

**User Story:** Как пользователь, я хочу, чтобы файлы `docs/` и `AGENTS.md` оставались человекочитаемым источником истины, а SML был их stateful-обёрткой и индексом, чтобы не терять контроль над памятью при проблемах со слоем.

#### Acceptance Criteria

1. THE Shared_Memory_Layer SHALL рассматривать File_Memory как авторитетный источник истины для Memory_Record, извлечённых из File_Memory.
2. WHEN содержимое любого файла из File_Memory изменяется, THE Shared_Memory_Layer SHALL обновить соответствующие Memory_Record в течение 60 секунд после фиксации изменения файловой системой.
3. IF Memory_Record в Shared_Memory_Layer расходится с содержимым соответствующего файла File_Memory, THEN THE Shared_Memory_Layer SHALL считать версию File_Memory истинной, привести Memory_Record в соответствие с File_Memory и зафиксировать факт расхождения в журнале синхронизации.
4. THE Shared_Memory_Layer SHALL сохранять для каждого извлечённого из File_Memory Memory_Record ссылку на исходный файл и диапазон строк в формате "путь_к_файлу:начальная_строка-конечная_строка".
5. THE Shared_Memory_Layer SHALL сохранять возможность записи Agent в файлы каталогов `docs/agent-log/`, `docs/decisions.md`, `docs/tasks.md` и `docs/memory/layers/` без изменения существующего формата этих файлов и без блокировок со стороны Shared_Memory_Layer длительностью более 5 секунд на операцию записи.
6. IF Shared_Memory_Layer недоступен или возвращает ошибку при чтении или записи, THEN THE Shared_Memory_Layer SHALL позволить Agent продолжать запись в File_Memory напрямую и повторно индексировать изменённые файлы в течение 60 секунд после восстановления доступности.
7. WHEN Agent создаёт или изменяет файл в каталогах `docs/agent-log/`, `docs/decisions.md`, `docs/tasks.md` или `docs/memory/layers/`, THE Shared_Memory_Layer SHALL создать или обновить соответствующие Memory_Record с сохранением ссылки на исходный файл и диапазон строк.

### Требование 9: Русский язык интерфейса и данных

**User Story:** Как русскоязычный пользователь, я хочу, чтобы SML принимал, хранил и возвращал данные на русском без потерь, чтобы соответствовать правилам проекта.

#### Acceptance Criteria

1. WHEN Agent передаёт текстовые поля Memory_Record в кодировке UTF-8 на русском языке, THE Shared_Memory_Layer SHALL сохранять исходные символы кириллицы без экранирования, транслитерации, перекодирования и замены символов, обеспечивая побайтовое равенство при последующем чтении.
2. WHEN Agent запрашивает Memory_Record через MCP, THE Shared_Memory_Layer SHALL возвращать текстовые поля в кодировке UTF-8 с сохранением кириллицы, пробелов и знаков пунктуации идентично сохранённому значению.
3. IF Shared_Memory_Layer формирует сообщение об ошибке MCP, THEN THE Shared_Memory_Layer SHALL возвращать текст сообщения на русском языке в кодировке UTF-8 с указанием причины ошибки и идентификатора операции.
4. WHEN Semantic_Query задан на русском языке длиной от 1 до 1000 символов в кодировке UTF-8, THE Shared_Memory_Layer SHALL обрабатывать его без предварительного перевода на другой язык и сопоставлять с Memory_Record на русском языке.
5. IF входной текст Memory_Record или Semantic_Query не является валидной последовательностью UTF-8, THEN THE Shared_Memory_Layer SHALL отклонять запрос с сообщением об ошибке на русском языке и не сохранять частичные данные.

### Требование 10: Безопасность и журналирование операций

**User Story:** Как владелец инфраструктуры, я хочу, чтобы SML не накапливал секреты и вёл журнал всех операций, чтобы соответствовать правилам безопасности `AGENTS.md`.

#### Acceptance Criteria

1. WHEN Agent выполняет операцию чтения или записи над Memory_Record, THE Shared_Memory_Layer SHALL добавить в Operation_Log запись, содержащую имя Agent, метку времени в формате ISO 8601 с точностью до секунды и часовым поясом UTC, идентификатор Memory_Record, тип операции из множества {read, write, delete}, и признак результата из множества {success, rejected, error}, в срок не более 1 секунды после завершения операции.
2. IF входной текст Memory_Record при операции записи содержит значение, распознанное как секрет (API-ключ, токен доступа или пароль) по набору проверок, включающему совпадение с известными префиксами секретов и обнаружение строк высокой энтропии длиной от 20 до 200 символов, THEN THE Shared_Memory_Layer SHALL отклонить операцию записи, не сохранять содержимое Memory_Record, вернуть вызывающему Agent ошибку с указанием категории причины отклонения без раскрытия самого значения секрета, и зафиксировать в Operation_Log запись с признаком результата rejected и категорией причины.
3. THE Shared_Memory_Layer SHALL хранить Operation_Log на локальном диске в каталоге проекта, обеспечивать доступность файла Operation_Log для чтения средствами операционной системы без запуска Shared_Memory_Layer, сохранять записи Operation_Log в режиме только добавления (append-only) без перезаписи существующих записей, и удерживать записи Operation_Log не менее 30 календарных дней с момента их создания.
4. THE Shared_Memory_Layer SHALL выполнять все операции над Memory_Record и Operation_Log локально в пределах рабочей станции, и SHALL NOT инициировать исходящие сетевые соединения, передающие содержимое Memory_Record или Operation_Log на адреса за пределами loopback-интерфейса.

### Требование 11: Производительность чтения при типичных объёмах

**User Story:** Как пользователь, я хочу, чтобы агенты получали ответы от SML быстро, чтобы не терять время на ожидание в каждом сеансе.

#### Acceptance Criteria

1. WHEN база Shared_Memory_Layer содержит не более Typical_Volume Memory_Record и Agent выполняет Semantic_Query без фильтров на локальной машине, THE Shared_Memory_Layer SHALL вернуть до 20 первых результатов за время не более 500 миллисекунд, измеренное от момента получения запроса до момента отправки полного ответа.
2. WHEN Agent выполняет операцию получения стартового контекстного пакета и база Shared_Memory_Layer содержит не более Typical_Volume Memory_Record на локальной машине, THE Shared_Memory_Layer SHALL вернуть ответ за время не более 1000 миллисекунд, измеренное от момента получения запроса до момента отправки полного ответа.
3. WHEN Agent выполняет операцию чтения Memory_Record по идентификатору и база Shared_Memory_Layer содержит не более Typical_Volume Memory_Record на локальной машине, THE Shared_Memory_Layer SHALL вернуть ответ за время не более 200 миллисекунд, измеренное от момента получения запроса до момента отправки полного ответа.
4. IF Semantic_Query от Agent не возвращается в пределах установленного лимита времени для соответствующей операции, THEN THE Shared_Memory_Layer SHALL завершить операцию с признаком превышения лимита времени и сохранить целостность базы Memory_Record без частичной записи.
5. IF база Shared_Memory_Layer содержит более Typical_Volume Memory_Record, THEN THE Shared_Memory_Layer SHALL выполнить операцию чтения и вернуть признак выхода за пределы Typical_Volume без гарантии указанных лимитов времени.

### Требование 12: Fallback при недоступности слоя

**User Story:** Как пользователь, я хочу, чтобы агенты продолжали работать с File_Memory, если SML недоступен, чтобы отказ слоя не блокировал задачи.

#### Acceptance Criteria

1. IF MCP-клиент Agent не может установить соединение с Shared_Memory_Layer в течение 5 секунд с момента инициирования подключения, THEN THE Agent SHALL перейти в Fallback_Mode, зафиксировать факт перехода в журнале Session и продолжить работу с File_Memory без прерывания текущей задачи.
2. WHEN Agent работает в Fallback_Mode, THE Agent SHALL читать стартовый контекст по документированному списку файлов File_Memory из `docs/context-index.md` и, IF указанный список файлов недоступен или пуст, THEN THE Agent SHALL зафиксировать ошибку чтения стартового контекста и продолжить работу с пустым стартовым контекстом.
3. WHEN Shared_Memory_Layer снова становится доступен в течение той же Session и операция проверки доступности возвращает успешный статус в течение 5 секунд, THE Agent SHALL восстановить подключение к Shared_Memory_Layer, выйти из Fallback_Mode и использовать Shared_Memory_Layer для последующих операций в рамках Session.
4. THE Shared_Memory_Layer SHALL предоставлять операцию проверки доступности, возвращающую в течение 2 секунд статус доступности и версию сервиса, а IF операция проверки доступности не возвращает результат в течение 2 секунд или возвращает статус ошибки, THEN THE Agent SHALL считать Shared_Memory_Layer недоступным и оставаться в Fallback_Mode.
5. WHILE Agent находится в Fallback_Mode, THE Agent SHALL периодически выполнять операцию проверки доступности Shared_Memory_Layer с интервалом 30 секунд до момента восстановления подключения или завершения Session.

### Требование 13: Интеграция с существующими протоколами агентов

**User Story:** Как пользователь, я хочу, чтобы SML работал внутри уже действующих протоколов `AGENTS.md`, `docs/memory-autoprotocol.md` и формата `docs/agent-log/`, чтобы новый слой не ломал текущие привычки агентов.

#### Acceptance Criteria

1. WHEN Agent вызывает операцию поиска похожего контекста с непустым поисковым запросом длиной от 1 до 512 символов, THE Shared_Memory_Layer SHALL возвращать список Memory_Record, упорядоченный по убыванию релевантности, содержащий от 0 до 20 элементов, достаточный для выполнения шагов автопротокола из `docs/memory-autoprotocol.md`.
2. WHEN Agent вызывает операцию добавления записи журнала агента с заполненными полями даты, имени агента, исходного запроса и результата, THE Shared_Memory_Layer SHALL создавать Memory_Record типа «запись журнала» и соответствующий файл в каталоге `docs/agent-log/` в течение 5 секунд.
3. WHEN Agent вызывает операцию добавления решения с заполненными полями заголовка, контекста и итогового решения, THE Shared_Memory_Layer SHALL создавать Memory_Record типа «решение» и добавлять запись в `docs/decisions.md` в течение 5 секунд, сохраняя ранее существующее содержимое файла без изменений.
4. WHEN Agent вызывает операцию сборки контекстного пакета, THE Shared_Memory_Layer SHALL обновлять файл `docs/context-packs/context-pack-latest.md` в формате, совместимом с текущей автоматизацией `tools/watch-memory.ps1`, в течение 10 секунд.
5. IF операция добавления записи журнала, добавления решения или сборки контекстного пакета завершается ошибкой из-за недоступности файловой системы или нарушения обязательных полей, THEN THE Shared_Memory_Layer SHALL возвращать Agent сообщение об ошибке с указанием причины отказа и SHALL сохранять целевые файлы `docs/agent-log/`, `docs/decisions.md` и `docs/context-packs/context-pack-latest.md` в состоянии, предшествовавшем вызову операции.

### Требование 14: Совместимость с фоновой автоматизацией памяти

**User Story:** Как пользователь, я хочу, чтобы фоновый наблюдатель `tools/watch-memory.ps1` и задача Windows Task Scheduler `Aion File Memory Auto` продолжали пересобирать контекстный пакет после включения SML, чтобы не терять существующую автоматизацию.

#### Acceptance Criteria

1. WHILE процесс `tools/watch-memory.ps1` выполняет чтение или запись файла `docs/context-packs/context-pack-latest.md` или файлов File_Memory, THE Shared_Memory_Layer SHALL выполнять операции чтения и записи над теми же файлами без взаимных блокировок, удерживая каждую собственную файловую блокировку не дольше 5 секунд и повторяя попытку до 3 раз при конфликте доступа.
2. WHEN `tools/watch-memory.ps1` завершает запись файла `docs/context-packs/context-pack-latest.md`, THE Shared_Memory_Layer SHALL обновить связанные Memory_Record в течение 60 секунд после фиксации записи и зафиксировать в журнале время начала и окончания обновления.
3. IF обновление связанных Memory_Record после записи `docs/context-packs/context-pack-latest.md` не завершилось в течение 60 секунд или завершилось с ошибкой, THEN THE Shared_Memory_Layer SHALL сохранить предыдущее состояние Memory_Record без изменений, зарегистрировать сообщение об ошибке совместимости с фоновой автоматизацией и повторить попытку обновления не более 3 раз.
4. THE Shared_Memory_Layer SHALL запускаться как локальный процесс на Windows с PowerShell 7 без обязательных сетевых или облачных зависимостей и SHALL сохранять работоспособность чтения и записи Memory_Record при отсутствии подключения к сети.
