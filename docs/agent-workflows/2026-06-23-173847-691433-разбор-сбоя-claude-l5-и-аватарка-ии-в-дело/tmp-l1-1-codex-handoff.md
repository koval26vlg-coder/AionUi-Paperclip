## Что было сделано

Проверены факты по сбою Claude CLI. Первичная команда L5 зависла/не вернула output за 3 минуты. Дополнительные проверки показали:

- `claude --version` работает;
- некорректная форма `claude -p ... --tools "" "prompt"` может вернуть `Input must be provided...`;
- слишком низкий `--max-budget-usd 0.05` дает `Exceeded USD budget`;
- `claude doctor` в этой среде может зависать;
- корректный путь `prompt | claude -p --model haiku --allowedTools Read --max-budget-usd 0.30` успешно вернул `READ_OK` и L5-решение `decision=approve`.

## На чем основан вывод

Основание: вывод команд диагностики Claude CLI, наличие зависших `claude.exe`, их остановка, успешный короткий read-only запуск на `haiku`.

## Что получилось хорошо

Сбой локализован не как полная поломка Claude CLI, а как сочетание тяжелого/неудачно сформированного L5 print-run и бюджетно-модельных ограничений. Найден рабочий безопасный рецепт.

## Что требует доработки

Для future L5 automation стоит использовать stdin-prompt, явный `--model haiku` для короткой проверки, `--allowedTools Read`, разумный бюджет и внешний timeout. `doctor` не использовать как быстрый health-check внутри workflow.

## Какие есть риски

Дефолтная модель может быть дороже или дольше отвечать; `doctor` может стартовать MCP/health checks и зависать. Низкий бюджет маскирует рабочий CLI под ошибку.

## Что нельзя потерять/исказить дальше

Не утверждать, что Claude CLI не работает. Рабочая формулировка: CLI работает при корректном коротком запуске; зависание было в конкретном L5-run сценарии.
