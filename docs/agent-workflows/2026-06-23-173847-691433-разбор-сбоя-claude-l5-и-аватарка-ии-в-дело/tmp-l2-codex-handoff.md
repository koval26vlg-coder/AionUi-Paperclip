## Что было сделано

Инженерно проверен вывод: воспроизведены три режима Claude CLI. Ошибки и рабочий путь разделены:

1. Плохая/неустойчивая форма команды с `--tools` и позиционным prompt может неверно распарсить input.
2. `--max-budget-usd 0.05` слишком низкий даже для короткой проверки и дает budget error.
3. `doctor` не подходит для короткой L5-проверки, так как зависал по timeout.
4. Рабочий путь: stdin prompt + `-p` + `--model haiku` + `--allowedTools Read` + бюджет `0.20-0.30`.

## На чем основан вывод

Основание: реальные PowerShell команды с таймаутами, завершение зависших `claude.exe`, успешные ответы `ОК`, `READ_OK`, `decision=approve`.

## Что получилось хорошо

Есть проверенный operational workaround для следующих workflow: не запускать тяжелый Claude L5 по умолчанию, а делать короткий scoped review.

## Что требует доработки

В идеале добавить wrapper для Claude L5 review, который принимает prompt через stdin, задает модель/бюджет/allowedTools, пишет stdout/stderr и жестко убивает процесс после timeout.

## Какие есть риски

Без wrapper агенты могут повторить старый pattern и снова получить зависший процесс или пустой результат.

## Что нельзя потерять/исказить дальше

Причина не в Telegram и не в создании канала. Это локальный orchestration issue вокруг Claude Code CLI print-mode.
