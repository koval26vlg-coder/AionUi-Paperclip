## Что требует доработки
- Перед тем как просить пользователя подтвердить фактический 6-часовой сбор (`-ConfirmedLongRun`), необходимо:
  1. Реализовать префлайт-проверку (preflight check) в коллекторе, которая принудительно завершит сбор (early exit) через 30-60 минут, если плотность записей (rows per minute/cycle) окажется ниже минимально допустимого порога.
  2. Добавить сброс флага/состояния `next_step_after_ready` или явную блокировку чтения старого забракованного датасета в логике запуска постпроцессинга, чтобы предотвратить stale next_step.
  3. Внедрить автоматическую валидацию структуры собираемых JSON/CSV файлов (схемы вебсокетов) сразу после тестовой минуты сбора.

## Какие есть риски
- Запуск долгосрочного сбора данных без реальной визуализации прогресса или без интерактивного терминала (нарушение Visible Run Rule).
- Риск сетевого сбоя или блокировки вебсокет-соединения со стороны биржи при непрерывном 6-часовом подключении, что снова приведет к разреженному набору данных.
- Нестабильность структуры/схемы данных вебсокетов при обновлении API бирж.

## Что нельзя потерять/исказить дальше
- Блокировку любых бэктестов или симуляций до подтверждения качества нового датасета (прохождение guard по строкам за цикл).
- Запрет на использование live API-ключей, реальных ордеров и маржинального плеча.
- Обязательное явное подтверждение пользователя для запуска долгоживущих процессов.
