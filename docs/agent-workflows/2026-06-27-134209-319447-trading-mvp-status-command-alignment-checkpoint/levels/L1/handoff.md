## Что было сделано
Проведен анализ предоставленного пакета ревью для задачи уровня L1 (research review). Проверена степень риска наличия устаревшей legacy-переменной/команды `visible_collect_command` в `tools/trading_goal_status.ps1`, указывающей на сбор фандинга в условиях блокировки `funding_blocked_by_swarm=true`. Оценена необходимость выравнивания вывода статуса под защищенный WS-путь сбора до подтвержденного длительного запуска.

## На чем основан вывод
- Существующий датасет фандинга `funding_collect_7d_spotliq_visible_20260617_185732` заблокирован из-за несоответствия критериям качества данных (`data_quality:min_min_rows_per_cycle`). Любое выполнение шагов фандинга запрещено.
- Наличие в `tools/trading_goal_status.ps1` устаревшей команды `visible_collect_command`, указывающей на сбор фандинга при `funding_blocked_by_swarm=true`, создает реальный риск случайного запуска заблокированного или некорректного процесса автоматизацией или оператором.
- `trading_next_goal_step.ps1` в данный момент корректно ссылается на безопасный WS-план сбора (`start_ws_collect_visible.ps1 -Hours 6 -PlanOnly`). Статусные выходы должны соответствовать этому пути.

## Что получилось хорошо
- Логика блокировки фандинга из-за качества данных (`funding_blocked_by_swarm=true`) надежно предотвращает запуски этапов ранжирования и бэктестинга.
- Использование безопасного режима `-PlanOnly` для шага WS-сбора минимизирует вероятность несанкционированного старта до получения флага `-ConfirmedLongRun`.

## Что требует доработки
- Требуется скорректировать `tools/trading_goal_status.ps1` (и связанные скрипты): при активном флаге блокировки `funding_blocked_by_swarm=true` значение `visible_collect_command` не должно указывать на сбор фандинга. Команду необходимо направить на защищенный WS-путь либо пометить как `deprecated`/`blocked`.

## Какие есть риски
- Риск запуска некорректного/неполного сбора фандинга в обход guard review, если legacy-команда `visible_collect_command` будет выполнена автоматикой.
- Нарушение Active Run Gate Rule и Visible Run Rule при отсутствии должной блокировки/перенаправления в статусных файлах.

## Что нельзя потерять/исказить дальше
- Ограничение на проведение транзакций (research-only, отсутствие API-ключей, плеча и реальных сделок).
- Требование об обязательном подтверждении пользователя и наличии флага `-ConfirmedLongRun` для любых фактических долгосрочных запусков сбора.
- Причину блокировки фандинга (`data_quality:min_min_rows_per_cycle`).
