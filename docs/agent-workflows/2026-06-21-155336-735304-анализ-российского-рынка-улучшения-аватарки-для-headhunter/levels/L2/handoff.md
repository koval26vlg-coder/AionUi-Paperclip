## Что было сделано

Проведена инженерно-продуктовая проверка L1: можно ли из идеи сделать конкурентный MVP и где слабые места.

Оператор: Codex. Внешний runtime Antigravity CLI не вызывался; handoff оформлен как delegated executor run через workflow.

## На чем основан вывод

На L1 handoff, brief, events, официальных материалах hh.ru и конкурентном скане:
- hh.ru monthly reports показывают рост конкуренции среди соискателей: https://hh.ru/article/26641
- hh.ru март 2026: hh.индекс 11,4, вакансии -27%, резюме +41% год к году: https://stats.hh.ru/api/v1/monthly-report/f-983c2f9a-0568-4976-8b30-83bce102140b
- Generic competitors закрывают фото: AIQA, Fotor, HeadshotMaster, remove.bg, Canva.
- HH-specific competitors сильнее сфокусированы на вакансии, резюме, cover letter и отклике: HHBro и похожие расширения.

## Что получилось хорошо

Продукт можно реализовать быстро, если не начинать с собственной генеративной модели. MVP может быть orchestration-продуктом:
- загрузка фото;
- анализ качества и делового соответствия;
- генерация инструкций/вариантов обработки через внешний image API;
- проверка резюме и вакансии;
- итоговый "пакет отклика".

## Что требует доработки

Нужно сменить позиционирование:
- плохо: "улучшатель аватарки для HH";
- лучше: "HH Resume Booster: фото + резюме + сопроводительное письмо под вакансию";
- еще лучше для теста спроса: "Проверка резюме перед откликом: что мешает получить приглашение".

Порог MVP:
- не делать интеграцию с hh.ru API на старте;
- не делать автоотклики;
- работать через copy/paste или браузерное расширение без нарушения правил платформы;
- не хранить фото дольше необходимого;
- дать понятный before/after и чеклист.

## Какие есть риски

- Юридический риск персональных данных: фото, резюме, контакты, история работы.
- Платформенный риск hh.ru: автоматизация действий на сайте может нарушать правила.
- Риск недоказанного value: "красивее" не равно "больше приглашений".
- Технический риск качества AI-фото: артефакты лица, несоответствие реальной внешности.

## Что нельзя потерять/исказить дальше

Инженерный вывод: не строить кастомную генерацию на старте. Проверять спрос через легкий MVP, где фото - один модуль, а основная ценность - практическая подготовка резюме и отклика.
