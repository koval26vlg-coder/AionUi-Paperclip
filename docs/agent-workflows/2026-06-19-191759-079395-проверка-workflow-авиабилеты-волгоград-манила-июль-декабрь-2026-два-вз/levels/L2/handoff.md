## Что было сделано

L2 Antigravity CLI выполнен как engineering review gate после revision. Запуск был сделан в sandbox-режиме через `agy --sandbox --print`, вывод восстановлен из Antigravity conversation DB, чтобы агент не менял файлы workflow самостоятельно.

Проверены применимость L1-выводов, ограничения источников, риск ошибочного L2 handoff и пригодность результата для передачи на Codex L3.

## Что получилось хорошо

- L1 сохранил исходные критерии и не превратил задачу в общий поиск без дат/пассажиров.
- По двум календарным поверхностям Aviasales топ-варианты согласуются с минимальным расхождением около 1 RUB.
- Deep links для ручной проверки построены сразу под двух пассажиров.
- После найденной ошибки orchestration включен anti-distortion механизм: L2 был возвращен на revision с `disagreement.md`, а не silently overwritten.

## Что требует доработки

- Нужна ручная или браузерная проверка точной выдачи Aviasales по top-URL, потому что API exact search недоступен из окружения.
- Нужны детали маршрута: авиакомпании, пересадки, длительность, багаж, аэропорты пересадки, правила тарифа.
- Желательно сверить топ-даты с альтернативным агрегатором, если задача будет переходить от теста workflow к покупке.
- Нельзя считать покрытие всего периода полным: календарь дал много пустых дат, поэтому осень/зима могут быть недопредставлены.

## Какие есть риски

- Calendar fare может быть кэшированным и устаревшим.
- Цена для двух взрослых рассчитана как `calendar_price * 2`; если в минимальном fare bucket осталось одно место, checkout будет дороже.
- Маршрут VOG -> MNL почти наверняка требует стыковок; возможны длительные ожидания, самостоятельный перенос багажа или транзитные требования.
- Июль на Филиппинах попадает в сезон муссонов/тайфунов; дешевизна дат может быть связана с сезонным спросом и погодным риском.
- Antigravity runtime требует sandbox/no-write для таких workflow, иначе он может сам двигать state.
