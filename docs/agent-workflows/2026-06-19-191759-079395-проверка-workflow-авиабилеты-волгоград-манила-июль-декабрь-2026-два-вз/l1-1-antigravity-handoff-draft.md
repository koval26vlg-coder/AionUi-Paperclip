- Главный минимум найден и подтвержден двумя календарными поверхностями с расхождением только в 1 RUB между `api_v6_calendar` и `city_calendar_by_departure`.
- Для топ-кандидатов есть прямые Aviasales search URLs с суффиксом `2` для двух пассажиров.
- Ограничение calendar-vs-checkout явно зафиксировано, а не скрыто.

## Что требует доработки

- Нужно проверить топ-кандидаты в точной UI/checkout-выдаче Aviasales или альтернативном агрегаторе, потому что exact search API в текущем окружении возвращал 403.
- Нужно получить детали рейса: перевозчики, пересадки, длительность маршрута, багаж, аэропорты пересадки и правила тарифа.
- Нужно проверить, не изменится ли цена при покупке именно на 2 взрослых одновременно.
- Из-за пустых календарных ответов по большинству дат нельзя заявлять, что это абсолютный минимум по рынку.

## Какие есть риски

- Calendar API является предварительным и кэшированным источником.
- `estimated_two_adults_rub` получен линейным умножением calendar price на 2, а не подтверждением двух мест в одном классе бронирования.
- Цена динамическая и может измениться после перехода на страницу продавца.
- Дешевые маршруты VOG -> MNL могут иметь сложные стыковки и багажные ограничения.
- Antigravity runtime восстановил ответ из DB fallback, что подтверждает работу wrapper, но требует сохранять фактические изменения только через workflow CLI.
