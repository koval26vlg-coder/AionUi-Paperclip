# L1 Handoff: Android VPN diagnostics

## Что было сделано
Создан workflow Роя для Android VPN. Выполнена попытка получить L1 через Antigravity CLI, но `agy --print` снова вернул пустой stdout без восстановленного DB-ответа. После этого Codex провел локальную и серверную диагностику.

## На чем основан вывод
Проверены локальные Android-профили WireGuard/OpenVPN, сервер WireGuard/OpenVPN по SSH, `wg show`, `wg0.conf`, sysctl forwarding, NAT/iptables/nft, слушающие порты и короткий tcpdump по `10.8.0.3`.

## Что получилось хорошо
Серверная сторона WireGuard работает: `wg0` активен, `net.ipv4.ip_forward=1`, NAT для `10.8.0.0/24` включен, peer `10.8.0.3/32` имеет свежий handshake и мегабайты transfer в обе стороны. Tcpdump показал реальный трафик телефона через `wg0` к внешним HTTPS/UDP адресам и ответы обратно. С VPS доступны `api.ipify.org`, YouTube, Google, Telegram и Telegram API. Создан чистый профиль `C:\Users\koval\Desktop\zlwg.conf`, его public key совпадает с серверным Android peer.

## Что требует доработки
Пользователь должен импортировать именно `zlwg.conf` и удалить/отключить старые Android VPN-профили `zlwg1`, `zlwg2`, `zlwg3`, `android-wg1`. На телефоне нужно проверить, что WireGuard не исключает браузер/Telegram/YouTube из VPN, `Private DNS` выключен, старое VPN-приложение не назначено как Always-on VPN.

## Какие есть риски
Файл `zlwg.conf` содержит приватный ключ, его нельзя публиковать. Не нужно менять серверный NAT вслепую, потому что текущие счетчики и tcpdump показывают рабочую серверную маршрутизацию. Не нужно автоматически трогать Windows Cisco/Happ.

## Что нельзя потерять/исказить дальше
Главный вывод: проблема больше не выглядит серверной. Если телефон продолжает видеть внешний IP `178.155.115.116`, это означает обход WireGuard на самом телефоне или использование не того профиля. Правильный IP после включения `zlwg`: `147.90.11.165`.
