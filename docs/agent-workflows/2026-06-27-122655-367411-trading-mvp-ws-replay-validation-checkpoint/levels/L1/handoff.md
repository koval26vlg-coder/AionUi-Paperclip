## Что было сделано
- Проведен независимый анализ предложенного Codex плана по внедрению обертки валидации и симуляции (replay) для данных WebSocket в рамках торгового MVP (`trading_mvp`).
- Сопоставлены факты текущего состояния: отклонение собранного набора данных по фандингу из-за низкого качества (data-quality rejection: `min_rows_per_cycle = 9` при пороге `20`) и отсутствие явного одобрения пользователя на запуск нового 6-часового сбора сырых WS-данных.
- Проверена применимость внедрения `run_ws_replay_validation_visible.ps1` перед проведением реального тестирования на исторических данных (backtest/replay) или запуска живых стратегий.

## На чем основан вывод
- Текущие собранные данные не прошли проверку качества (`blocked_by_data_quality`), поэтому любые попытки рассчитать ранги, запустить бэктесты или paper-forward симуляции на них запрещены и приведут к недостоверным результатам.
- Запуск нового 6-часового сбора данных WebSocket требует предварительного явного подтверждения от пользователя.
- Наличие защитной обертки для постобработки `run_ws_postprocess_visible.ps1` требует следующего логического шага: защищенного воспроизведения (guarded replay) и валидации результатов постобработки перед тем, как переходить к оптимизации параметров сетки (grid) и реальной симуляции.
- План Codex содержит необходимые меры безопасности: режим PlanOnly, проверку статуса активности сбора/постобработки (отказ при RUNNING или STOPPED_INCOMPLETE), явный путь к артефакту постобработки (`PostprocessPath`) и ворота допуска (`replay_allowed` gate), активируемые только при наличии флага подтвержденного исследовательского запуска (`ConfirmedResearchRun`).

## Что получилось хорошо
- Предложенный Codex план предотвращает случайный запуск симуляций на неполных или некачественных данных.
- Механизм блокировок на основе статуса активности (`RUNNING` / `STOPPED_INCOMPLETE`) исключает конфликты и гонки ресурсов во время сбора данных.
- Введение ворот допуска (`replay_allowed` gate) страхует от невалидных бэктестов при плохом качестве постобработанных файлов.

## Что требует доработки
- Требуется убедиться, что ворота допуска (`replay_allowed` gate) жестко привязаны к метрике качества данных (например, порогу `min_rows_per_cycle >= 20`), зафиксированному при проверке качества постобработки.
- Скрипт `run_ws_replay_validation_visible.ps1` должен явно логировать причины отказа (например, "rejecting replay: ws_postprocess artifact failed data-quality gate").

## Какие есть риски
- Риск запуска симуляции на устаревшем или нерелевантном датасете, если `PostprocessPath` указан неверно или ведет к старому запуску.
- Риск обхода ограничений при некорректной передаче флага `ConfirmedResearchRun` без прохождения проверки качества на этапе постобработки.

## Что нельзя потерять/исказить дальше
- Обязательную проверку статуса активного сбора через скрипт `check_active_run_gate.ps1`.
- Запрет на запуск нового 6-часового WS collect без явного подтверждения пользователя.
- Полное отсутствие живых ордеров (live orders), использования ключей API и кредитного плеча (leverage/margin).
