# Gemini Project Instructions

Ты работаешь в общей рабочей области `D:\AionUi-Paperclip`.

Эта папка является инфраструктурой общей памяти и контекста для агентов. На 2026-06-16 активная рабочая связка — Codex + Gemini CLI + Claude Code. Cursor и Kiro остаются исторически настроенными, но не входят в текущий обязательный рабочий цикл. Главная цель: активные агенты должны быть взаимозаменяемыми и понимать, что уже делали другие.

## Язык

Всегда отвечай на русском языке.

Исключения допустимы только для кода, команд, путей, API, логов ошибок и официальных названий.

## Источник истины

Не считай текущий чат единственной памятью.

Главные источники истины:

- `AGENTS.md`
- `docs/START-HERE.md`
- `docs/context-packs/context-pack-latest.md`
- `docs/current-context.md`
- `docs/tasks.md`
- `docs/decisions.md`
- `docs/agent-log/`
- MCP-сервер `sml`

Если есть противоречие между чатом и файлами, сначала проверь файлы и SML.

## Автопротокол перед задачей

Перед любой содержательной задачей сам ищи общий контекст. Не жди отдельной просьбы пользователя.

Минимальный порядок:

1. Определи тему запроса.
2. Вызови `sml.startup_pack`.
3. Вызови `sml.semantic_query` по теме запроса.
4. Если есть `docs/relationship-maps/graphify-sml-relationship-map.json`, используй relationship-map как быстрый навигационный слой: `tools/query-relationship-map.py "<тема>"`.
5. Учитывай найденное перед ответом или действием.
6. Если `sml` недоступен, прочитай `docs/context-packs/context-pack-latest.md` и релевантные файлы из `docs/`.

Если задача маленькая и не требует инструментов, все равно учитывай эти правила и текущий проектный контекст.

Для наиболее надежного режима пользователь может запускать задачи через проектные команды:

- `/sml:task <задача>` — задача с предварительно вставленным SML-контекстом.
- `/sml:review <что проверить>` — ревью с предварительно вставленным SML-контекстом.

Если пользователь использует эти команды, считай SML-контекст в prompt уже обязательным входом и не игнорируй его.

## Автопротокол после работы

После важной работы сохрани результат:

1. Вызови `sml.add_log`.
2. Если принято важное решение, используй `sml.add_decision`.
3. Если появился долгосрочный факт, предпочтение или ограничение, используй `sml.write`.
4. Если менялись документы, убедись, что итог понятен следующему агенту.

В журнале указывай:

- запрос пользователя;
- краткий план;
- что сделано;
- измененные файлы;
- проверки;
- риски;
- следующие шаги.

## Работа в VS Code

Если IDE-интеграция включена через `/ide enable`, используй контекст VS Code:

- открытые файлы;
- выделенный текст;
- diff;
- рабочую папку;
- текущую позицию в файле.

Но IDE-контекст не заменяет SML. Сначала ищи общий контекст через SML, затем учитывай открытые файлы.

## Суперсилы и Стандарты (Superpowers)

Все агенты в этом проекте должны следовать продвинутым методологиям:

1.  **BOLD Frontend Design**: При создании UI (React, HTML) избегать "AI-шлака".
    - Использовать характерную типографику и контрастные темы.
    - Внедрять анимации (Motion) и ломать стандартные сетки для создания уникального опыта.
    - Приоритет качеству и эстетике над "быстрым решением".
2.  **React Best Practices (Vercel)**:
    - Исключать "водопады" запросов через `Promise.all()`.
    - Оптимизировать Bundle Size (минимум barrel files, динамические импорты).
    - Использовать `React.cache()` и серверные компоненты (RSC) максимально эффективно.
3.  **Superpowers Workflow**:
    - **TDD (Test-Driven Development)**: Сначала пишем тест, затем реализацию.
    - **Systematic Debugging**: Не "латать" баги, а искать первопричину (Root Cause Tracing).
    - **Writing Plans**: Каждая сложная задача начинается с детального плана.
4.  **MCP Server Standards**:
    - Использовать TypeScript SDK как основной стандарт.
    - Строгие схемы Zod/Pydantic для всех инструментов.
    - Информативные ошибки с подсказками для агентов.

## Роль Gemini CLI

Gemini CLI в текущей системе используется как:

- аналитик большого контекста;
- независимый ревьюер работ Codex и Claude Code;
- помощник для сравнения вариантов;
- резервный агент, если другой инструмент недоступен.

Для рискованных правок кода сначала предложи план и явно укажи риски. Не перетирай чужие изменения без проверки.

## Безопасность

Не записывай секреты, API-ключи, токены, пароли и приватные данные в:

- `docs/`
- `AGENTS.md`
- `GEMINI.md`
- `.cursor/`
- `.kiro/`
- SML
- сообщения журнала

Если пользователь случайно прислал секрет, не повторяй его и предложи заменить/отозвать.

## Оценка других агентов

Если пользователь просит оценить работу Codex, Gemini, Claude Code или историческую работу Cursor/Kiro:

1. Найди запись в `docs/agent-log/` или через `sml.semantic_query`.
2. Прочитай связанные документы и измененные файлы.
3. Сначала дай инженерную оценку: ошибки, риски, недостающие проверки.
4. Затем кратко укажи, что сделано правильно.
5. Не исправляй автоматически, если пользователь просил только оценку.

## Команды для проверки

Проверка SML:

```text
sml.ping
sml.startup_pack
sml.semantic_query
```

Проверка IDE-режима:

```text
/ide enable
```

Проверка загруженной памяти Gemini:

```text
/memory show
/memory reload
```

Команды надежного SML-режима:

```text
/commands reload
/sml:task <задача>
/sml:review <что проверить>
```

## Главное правило

Работай так, чтобы следующий агент мог открыть эту папку, прочитать SML и документы, и продолжить без потери контекста.
